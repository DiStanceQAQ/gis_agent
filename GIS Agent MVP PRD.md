### 1.1 文档目的

本文档用于定义 GIS Agent 的 MVP 范围、目标用户、核心流程、功能需求、算法规则、非功能要求、验收标准与上线边界，供产品、研发、设计与测试协同执行。

### 1.2 产品一句话定义

GIS Agent 是一个通过自然语言完成基础 GIS / 遥感科研工作流的智能助手，能够理解用户研究任务、自动选择合适遥感数据、执行 NDVI 计算、输出地图与 GeoTIFF，并对结果与方法进行解释。

### 1.3 本文档适用对象

- 产品经理
- 后端工程师
- GIS / 遥感工程师
- 前端工程师
- 测试工程师
- 设计师
- 创始团队 / 业务负责人

## 2. 项目背景

### 2.1 当前问题

目标用户在进行 GIS / 遥感科研时，普遍存在以下问题：

1. **不知道先用什么数据。**
   同一个任务可能可以使用 Sentinel-2、Landsat、HLS 等多种数据源，不同数据在空间分辨率、时间密度、历史连续性、云质量可用性上差异较大。对多数用户来说，选择过程本身就是门槛。
2. **会提问题，不会走完整流程。**
   很多用户知道自己想研究什么，例如“某地某季节植被状况”，但不知道如何把这个需求落成 AOI、时间窗、数据检索、云掩膜、指数计算、合成、出图等一整条处理链。
3. **GIS 软件操作门槛高。**
   即使具备基础科研能力，用户仍需要在多个工具之间来回切换：查数据目录、下载数据、处理栅格、做裁剪、出图、写方法说明，流程分散且重复。
4. **结果不易解释与复现。**
   很多自动化脚本能“跑出结果”，但不能说明“为什么选这个数据”“参数是什么”“为什么图上是这个样子”，也不能稳定地保存处理链路，导致结果难以复核和重用。

### 2.2 项目机会

如果能把“提需求—选数据—执行计算—出图—解释结果”串成一条可对话、可追踪、可复现的工作流，就能显著降低 GIS / 遥感分析门槛，让研究者把时间从重复劳动转移到问题本身。

### 2.3 为什么先做 MVP

本项目目标不是第一版就做成通用 GIS 平台，而是验证下面三件事：

1. 用户是否愿意用自然语言描述遥感分析任务。
2. 系统是否能在有限范围内稳定完成一条完整分析链路。
3. “自动推荐数据 + 自动执行 + 自动解释”是否明显优于传统脚本或纯聊天助手。

## 3. 产品目标与非目标

### 3.1 MVP 目标

MVP 需要在单一核心场景上达到“可用、可信、可复现”：

- 用户可以通过自然语言输入一个基础植被分析任务。
- 系统可以识别研究区域、时间范围、指数类型与输出需求。
- 系统可以在支持的数据源中自动搜索候选数据，并给出推荐理由。
- 系统可以稳定完成 NDVI 计算与时间窗口合成。
- 系统可以输出可读地图、GeoTIFF 和方法说明。
- 系统可以记录关键参数，支持复核与重跑。

### 3.2 MVP 成功标准

MVP 不要求覆盖所有 GIS 场景，但必须在预定义范围内做到以下结果：

- 一个普通用户无需手工处理原始遥感数据，也能完成一次标准 NDVI 分析。
- 结果不是“黑盒图像”，而是带有数据选择理由、方法说明和局限性提示的研究输出。
- 系统在固定测试集上具有稳定成功率，不依赖人工临时干预。

### 3.3 非目标

以下内容明确不纳入 MVP：

- 通用桌面 GIS 替代方案
- 复杂矢量编辑与图层管理
- 大规模高级空间统计分析
- 自动论文全文生成
- 多 agent 自主协作系统
- 用户自定义任意栅格公式引擎
- 所有遥感产品与所有指数的全覆盖
- 多人协作、权限管理、团队空间

------

## 4. 产品定位

### 4.1 产品定位

GIS Agent 的定位不是“会聊天的 GIS 教程”，也不是“脚本包装器”，而是：

> 一个以前台对话交互驱动、以后端标准化 GIS 工作流执行的研究助手。

### 4.2 价值主张

对用户来说，GIS Agent 的核心价值是：

- **降低门槛**：用户不必熟悉具体数据下载与处理链路。
- **节省时间**：减少检索数据、写脚本、做基础出图的重复劳动。
- **可解释**：告诉用户为什么选某种数据、方法是什么、结果意味着什么。
- **可复现**：记录关键参数与输出，便于复核和继续分析。

### 4.3 差异化原则

MVP 采用“workflow 为主、agent 为辅”的方式设计：

- agent 负责理解任务、比较方案、解释结果。
- workflow 负责执行标准化 GIS / 遥感处理链。
- 不允许大模型在运行时临时生成核心栅格算法替代固定工具链。

## 5. 目标用户与使用场景

### 5.1 目标用户

| 用户类型              | 特征                           | 主要痛点                       |
| --------------------- | ------------------------------ | ------------------------------ |
| GIS / 遥感初学者      | 会提出研究问题，不熟悉数据处理 | 不知道选什么数据，不会做预处理 |
| 科研研究生            | 会基础软件操作，但流程重复     | 想更快做出初稿图和方法说明     |
| 行业分析人员          | 不专门做遥感，但有区域分析需求 | 缺少稳定的分析模板，效率低     |
| 有 GIS 基础的研究人员 | 会做分析，但讨厌重复劳动       | 希望缩短从问题到结果的路径     |

### 5.2 核心用户故事

1. 作为一个研究生，我希望输入“帮我算 2024 年夏季某区域 NDVI”，系统能自动完成数据选择、计算与出图，而不需要我自己去找数据和写脚本。
2. 作为一个科研人员，我希望系统能解释为什么选 Sentinel-2 而不是 Landsat，这样我能判断结果是否适合我的研究。
3. 作为一个用户，我希望导出 GeoTIFF 和方法说明，继续在自己的 QGIS / 论文草稿里使用。
4. 作为一个分析人员，我希望失败时系统能告诉我为什么失败，以及下一步应该怎么调整，而不是只给我一个报错。

### 5.3 核心使用场景

#### 场景 A：基础 NDVI 分析

用户输入：
“帮我计算 2024 年 6 到 8 月北京西山的 NDVI，并导出一张图和 GeoTIFF。”

系统输出：

- 结构化任务参数
- 候选数据源和推荐理由
- NDVI 图
- GeoTIFF 下载
- 方法说明和结果解释

#### 场景 B：数据方案选择

用户输入：
“我想看某县 2023 年春季农作物长势，用什么数据更适合？”

系统输出：

- Sentinel-2 / Landsat / HLS 比较
- 推荐数据源
- 推荐逻辑解释
- 可继续执行的分析入口

#### 场景 C：基于结果继续追问

用户输入：
“把刚才的结果再导出 GeoTIFF。”
“换成 2023 年重算。”
“再解释一下为什么这个区域 NDVI 偏低。”

系统输出：

- 基于会话上下文继续执行
- 无需重新输入全部参数

## 6. MVP 范围定义

### 6.1 支持的输入

MVP 支持以下输入形式：

#### 区域输入

- 地名，如“北京西山”
- 行政区名称，如“某县”“某市”
- 上传 `geojson`
- 上传 `shp.zip`（压缩后的 shapefile）

#### 时间输入

- 某个月，如“2024 年 7 月”
- 某个季节，如“2024 年夏季”
- 某个连续时间范围，如“2024 年 6 到 8 月”
- 某一年内的指定区间

#### 任务输入

- 明确 NDVI 分析
- 植被长势类表述，系统可默认映射为 NDVI 并提示确认

#### 输出输入

- 地图
- GeoTIFF
- 方法说明
- 简短结果解释

### 6.2 支持的数据源

MVP 仅支持以下三类遥感数据源：

1. Sentinel-2
2. Landsat 8/9
3. HLS

### 6.3 支持的分析类型

MVP 仅承诺支持以下分析类型：

- NDVI 计算
- 时间窗口合成（单景、中位数合成、最大值合成）
- 基础区域统计（均值、最小值、最大值，可选）

### 6.4 支持的输出

MVP 需支持以下输出形式：

- PNG 专题图
- GeoTIFF 栅格文件
- 方法说明（Markdown 文本）
- 简要结果摘要（文本）

### 6.5 不支持的能力

以下能力即使有技术可行性，也不进入 MVP：

- NDWI / EVI / LST 等多个指数同时支持
- 复杂时序趋势分析
- 复杂行政区分组统计
- 动态在线编辑图层
- WebGIS 全功能交互
- 报告导出到 Word / PDF
- 多任务并发编排
- 用户上传任意脚本执行

### 6.6 范围边界与默认限制

为了保证 MVP 可控，设置以下默认边界：

- 单次 AOI 建议不超过 20,000 平方公里。
- 单次时间范围建议不超过 12 个月。
- 如果用户输入超大区域或超长时间，系统应提示当前版本性能边界。
- 如果用户只说“看植被情况”但未明确指标，默认使用 NDVI，并在输出中说明。
- 如果用户未指定输出，默认输出 PNG 地图 + 方法说明。

## 7. 产品原则

### 7.1 结果可追踪优先于表面智能

宁可少支持一些功能，也要保证每次结果都能说清楚：

- 用了什么数据
- 为什么选这个数据
- 参数是什么
- 失败时卡在了哪里

### 7.2 固定工具链优先于自由生成代码

大模型不负责临时发明遥感算法，只负责：

- 理解任务
- 选择方案
- 解释结果

真正的计算必须由可测试的固定工具完成。

### 7.3 先把一条核心链路做透

MVP 聚焦“NDVI + 自动选数据 + 自动出图”。
任何会显著拖慢开发周期、扩大测试面、削弱结果稳定性的功能，都不进入第一阶段。

------

## 8. 核心用户流程

### 8.1 主流程

```text
用户输入任务
-> 任务解析
-> AOI 标准化
-> 搜索候选数据
-> 数据推荐
-> 用户确认 / 系统默认确认
-> 执行下载与计算
-> 质量检查
-> 出图与导出
-> 结果解释与方法说明
```

### 8.2 详细流程说明

#### 步骤 1：用户输入任务

用户通过自然语言描述研究目标，例如：

> 帮我计算 2024 年 6 到 8 月北京西山的 NDVI，输出地图和 GeoTIFF。

系统需要从中提取：

- AOI：北京西山
- 时间：2024-06-01 至 2024-08-31
- 指标：NDVI
- 输出：地图、GeoTIFF

#### 步骤 2：任务解析

系统将自然语言转为结构化任务对象。
如关键参数缺失，则最多发起 1 次澄清。例如缺少时间时询问“你希望分析哪个时间范围？”

#### 步骤 3：AOI 标准化

系统将地名或上传文件解析为可计算 AOI，统一到标准 CRS，并生成：

- 边界几何
- 面积
- bbox
- 合法性校验结果

#### 步骤 4：搜索候选数据

系统根据 AOI、时间范围与任务目标，搜索 Sentinel-2、Landsat、HLS 的可用数据，输出：

- 可用场景数
- 基本质量指标
- 空间分辨率
- 时间覆盖情况

#### 步骤 5：数据推荐

系统比较候选数据，并给出推荐：

- 推荐数据源
- 推荐理由
- 备选方案
- 风险提示

#### 步骤 6：执行计算

系统执行固定工作流：

- 读取数据
- 云掩膜 / QA 过滤
- 裁剪 AOI
- 统一波段映射
- 计算 NDVI
- 根据时间范围做合成
- 导出栅格

#### 步骤 7：质量检查

系统在导出前检查：

- 有效像元比例
- AOI 覆盖情况
- NDVI 值域是否合理
- 是否出现大量 nodata

若不合格，进入 fallback 逻辑。

#### 步骤 8：出图与导出

系统生成：

- PNG 专题图
- GeoTIFF
- 方法说明
- 结果摘要

#### 步骤 9：结果解释

系统解释：

- 为什么选了该数据
- 图像表达了什么
- 哪些结果可能受云或时间窗口影响
- 下一步可以做什么

### 8.3 异常流程

#### 异常 A：区域无法识别

系统行为：

- 告知当前区域名称无法可靠解析
- 提示用户改用更明确地名或上传边界文件

#### 异常 B：时间缺失

系统行为：

- 提示用户补充时间范围
- 不直接进入搜索与计算

#### 异常 C：候选数据可用性很差

系统行为：

- 返回“该时段云量过高/覆盖不足”
- 自动尝试备选数据或合成策略
- 给出建议：放宽时间窗或更换数据源

#### 异常 D：计算结果大面积空值

系统行为：

- 尝试 fallback
- 若仍失败，则给出明确失败原因与下一步建议

## 9. 功能需求

------

### 9.1 模块一：任务解析

#### 9.1.1 功能目标

把用户自然语言解析成结构化 GIS 任务参数，作为后续工作流的统一输入。

#### 9.1.2 输入

- 用户自然语言文本
- 当前会话上下文（若存在）
- 上传文件元信息（若存在）

#### 9.1.3 输出

结构化 `TaskSpec`，至少包含：

- `aoi_input`
- `time_range`
- `analysis_type`
- `preferred_output`
- `user_priority`
- `need_confirmation`

#### 9.1.4 详细要求

1. 识别区域、时间、任务类型与输出需求。
2. 支持从上下文继承参数，例如“换成 2023 年”时不要求用户重新输入 AOI。
3. 当输入存在模糊表达时，优先使用保守解释，不擅自扩展范围。
4. 当任务目标为“植被情况”“植被长势”时，可默认映射为 NDVI，并在结果页说明。
5. 缺少关键参数时，最多追问 1 次。关键参数包括：
   - 区域
   - 时间范围
6. 如果区域和时间都具备，系统应直接推进，不要求用户做多余确认。

#### 9.1.5 默认策略

- 未指定输出：默认 `png_map + methods_text`
- 未指定分析指标但语义明显与植被相关：默认 `NDVI`
- 未指定优先级：默认 `balanced`

#### 9.1.6 验收标准

- 在测试集上，区域识别准确率不低于 85%。
- 时间范围识别准确率不低于 90%。
- 会话内参数继承正确率不低于 90%。

------

### 9.2 模块二：AOI 处理

#### 9.2.1 功能目标

将地名或用户上传边界转为可计算的标准化空间对象。

#### 9.2.2 输入

- 地名 / 行政区名
- `geojson`
- `shp.zip`

#### 9.2.3 输出

- 标准化几何对象
- bbox
- 面积
- AOI 元信息
- AOI 校验状态

#### 9.2.4 详细要求

1. 支持地名解析成边界。
2. 支持读取 `geojson` 与 `shp.zip`。
3. 自动检查几何是否合法。
4. 自动转换到系统标准 CRS。
5. 输出 AOI 面积与 bbox。
6. 当 AOI 超出 MVP 推荐范围时，给出性能与结果风险提示。

#### 9.2.5 边界处理规则

- 如果上传多个 feature，默认合并为一个研究区。
- 如果几何存在小错误且可修复，可自动修复并记录。
- 如果几何不可用，明确报错，不继续执行。

#### 9.2.6 验收标准

- 常见 `geojson` / `shp.zip` 可成功读取。
- 几何校验失败时有可理解错误信息。
- 标准化 AOI 可被后续数据搜索模块直接使用。

------

### 9.3 模块三：候选数据搜索

#### 9.3.1 功能目标

在支持的数据源范围内，根据 AOI 和时间范围找到可用遥感数据候选，并汇总用于比较的元信息。

#### 9.3.2 输入

- 标准化 AOI
- 时间范围
- 分析类型（NDVI）

#### 9.3.3 输出

候选数据列表，每个候选至少包含：

- 数据源名称
- 可用场景数
- 空间分辨率
- 时间分辨率描述
- 覆盖完整性
- 云量 / 质量摘要
- 是否适合当前任务

#### 9.3.4 详细要求

1. 至少同时搜索 Sentinel-2、Landsat 8/9、HLS。
2. 返回适合比较的摘要，而不是只返回原始 item 列表。
3. 搜索结果需可用于后续推荐模块直接打分。
4. 当某数据源没有结果时，也需返回“无可用数据”的明确信息。
5. 若用户上传 AOI 较小，仍需按真实覆盖和可用性搜索，不能因为区域小而跳过比较。

#### 9.3.5 默认搜索策略

- 时间范围内全部搜索
- 使用 AOI bbox 进行初筛
- 计算场景级别的可用性摘要
- 以“能用于 NDVI”的可用波段为前置条件

#### 9.3.6 验收标准

- 三个数据源都能返回统一格式的候选摘要。
- 当数据源完全不可用时，能返回结构化失败结果而非空白。
- 搜索输出可直接供推荐模块使用。

------

### 9.4 模块四：数据源推荐

#### 9.4.1 功能目标

在多个候选数据中，选择最适合当前任务的数据，并向用户说明理由。

#### 9.4.2 输入

- 候选数据摘要
- 任务目标
- 用户优先级（空间细节 / 时间密度 / 平衡）

#### 9.4.3 输出

- 推荐数据源
- 推荐理由
- 备选方案
- 风险提示

#### 9.4.4 推荐逻辑

MVP 采用规则+评分方式，不使用黑盒学习模型。
评分参考以下维度：

| 维度       | 说明                           | 权重 |
| ---------- | ------------------------------ | ---- |
| 空间匹配度 | 当前任务是否需要更高空间分辨率 | 35%  |
| 时间匹配度 | 时窗内可用场景数与时间密度     | 25%  |
| 质量可用性 | 云量、有效像元比例、覆盖完整性 | 25%  |
| 任务适配性 | 是否适合做当前 NDVI 分析       | 15%  |

#### 9.4.5 规则补充

- 用户强调“细节”“地块”“局部差异”时，Sentinel-2 优先级上升。
- 用户强调“多期”“连续性”“跨多年”时，Landsat / HLS 优先级上升。
- 当某数据源有效像元比例明显不足时，不应被推荐为首选。
- 若首选与备选差异不大，也要说明“两个方案都可行”。

#### 9.4.6 输出示例

系统应输出类似文本：

> 推荐使用 Sentinel-2。原因是当前任务关注局部山区植被差异，且 2024 年 6–8 月该区域可用场景数量足够，10m 级空间分辨率更适合观察地表细节。HLS 可作为备选方案，在需要更稳定时间序列时使用。

#### 9.4.7 验收标准

- 推荐结果包含“推荐 + 理由 + 备选”三部分。
- 在测试集上，专家主观认为合理的推荐比例不低于 80%。
- 当无明显优势方案时，系统不强行给出过度自信的结论。

------

### 9.5 模块五：栅格处理与 NDVI 计算

#### 9.5.1 功能目标

通过固定的可测试工具链，完成 NDVI 分析的核心执行流程。

#### 9.5.2 输入

- 推荐数据源
- AOI
- 时间范围
- 合成策略
- 默认参数

#### 9.5.3 输出

- NDVI 栅格
- 中间执行日志
- 统计摘要

#### 9.5.4 执行子步骤

1. 拉取候选数据。
2. 进行云掩膜 / QA 过滤。
3. 标准化波段命名为 `red` / `nir`。
4. 裁剪至 AOI。
5. 计算 NDVI：
   `NDVI = (NIR - Red) / (NIR + Red)`
6. 根据时间范围进行合成：
   - 单景
   - 中位数合成
   - 最大值合成
7. 生成最终栅格。

#### 9.5.5 默认策略

- 默认合成方式：中位数合成
- 默认云量阈值：20%
- 默认输出分辨率：继承推荐数据源的原生优先分辨率
- 默认 nodata：统一设置并写入输出元数据

#### 9.5.6 实现边界

- 不支持用户在对话中自定义任意公式。
- 不支持多个指数在一次任务中同时计算。
- 不支持复杂重采样选项暴露给普通用户。

#### 9.5.7 验收标准

- NDVI 输出值域主要落在 [-1, 1]。
- 结果可成功裁剪并导出到 AOI 范围。
- 相同输入参数应得到一致结果。
- 中间日志可定位失败步骤。

------

### 9.6 模块六：质量检查与失败恢复

#### 9.6.1 功能目标

在结果交付给用户前进行质量检查，避免明显不可用结果直接输出。

#### 9.6.2 输入

- 计算后的栅格
- 中间统计信息
- 数据源候选摘要

#### 9.6.3 输出

- 通过 / 不通过
- 质量检查项结果
- fallback 建议或执行结果

#### 9.6.4 检查项

1. AOI 覆盖率是否足够。
2. 有效像元比例是否达标。
3. NDVI 值域是否合理。
4. 是否出现大面积空值。
5. 输出 CRS / 分辨率 / bbox 是否正确。

#### 9.6.5 通过阈值建议

- AOI 覆盖率 ≥ 95%
- 有效像元比例 ≥ 70%
- NDVI 超范围像元比例 ≤ 1%
- 空值比例 ≤ 30%

#### 9.6.6 fallback 逻辑

若质量检查不通过，依次尝试：

1. 切换为备选数据源
2. 放宽时间范围并重新合成
3. 将合成方式从单景改为中位数合成
4. 明确告知当前条件下结果不可用

#### 9.6.7 用户可见输出

当系统进入 fallback 时，应明确告知：

> 当前首选数据在该时段有效像元不足，系统已自动尝试使用备选数据和时间窗口合成。如果你更关注单时相结果，建议缩小研究时间或更换研究区。

#### 9.6.8 验收标准

- 不可用结果不能直接作为成功输出返回。
- fallback 流程有明确日志。
- 用户能看到失败原因或自动调整说明。

### 9.8 模块八：结果解释与方法说明

#### 9.8.1 功能目标

将结果转为用户可理解、可复用的自然语言说明，而不是只返回文件。

#### 9.8.2 输出内容

结果页需至少包含四部分：

1. **数据源说明**
   为什么选该数据，备选是什么。
2. **方法说明**
   研究区、时间范围、数据来源、NDVI 公式、云处理与合成方式。
3. **结果解释**
   图上高值、低值的大致含义，适合非专家快速理解。
4. **局限性提示**
   云量、合成方式、时间窗、空间分辨率等对结果的影响。

#### 9.8.3 示例结构

系统输出应接近如下形式：

- 本次分析使用 Sentinel-2 数据，对 2024 年 6–8 月研究区影像进行了云掩膜和中位数合成。
- NDVI 反映植被活跃程度，图中高值区域通常对应林地或生长状况较好的植被覆盖区。
- 结果受季节合成和云量过滤影响，不能直接视为单日观测结果。

#### 9.8.4 验收标准

- 每次成功任务必须生成方法说明。
- 解释内容不得与实际执行参数不一致。
- 局限性不能缺失。

## 11. 数据与算法规则

### 11.1 数据源统一抽象

为了屏蔽不同数据源的差异，系统内部统一使用以下抽象字段：

- `red`
- `nir`
- `qa`
- `cloud_metric`
- `spatial_resolution`
- `temporal_density`
- `coverage_ratio`

各数据源适配器负责完成原始波段名到统一抽象字段的映射。

### 11.2 NDVI 计算规则

NDVI 统一按以下公式计算：

```text
NDVI = (NIR - Red) / (NIR + Red)
```

当分母为 0 或像元为无效值时，输出 nodata。

### 11.3 合成规则

MVP 支持三种合成策略：

- `single_scene`：单景
- `median`：中位数合成
- `max`：最大值合成

默认策略为 `median`。

### 11.4 数据推荐规则

推荐规则遵循以下原则：

- 任务需要更高空间细节时，优先高分辨率数据。
- 任务时窗较长、需要更稳妥时，优先可用场景更多的数据。
- 任务可用性差时，优先有效像元比例高的数据。
- 无明显优势时，允许输出两个可行方案，不强制单一答案。

### 11.5 质量控制规则

- 有效像元比例不足时，不直接返回结果。
- NDVI 值异常时，需要标记失败并提示排查。
- AOI 覆盖不足时，不应作为成功结果。

### 11.6 默认参数

| 参数         | 默认值            |
| ------------ | ----------------- |
| 云量阈值     | 20%               |
| 合成策略     | 中位数合成        |
| 输出格式     | PNG + 方法说明    |
| 结果优先级   | 平衡              |
| 失败重试次数 | 1 次自动 fallback |

## 18. 风险与应对

### 18.1 数据可用性风险

风险：
某些区域或时间段数据云量过高、覆盖不足，导致结果不可用。

应对：

- 设计 fallback 流程
- 在产品中明确局限性
- 建立“失败也要可解释”的机制

### 18.2 区域解析风险

风险：
地名解析不精确，导致研究区错位。

应对：

- 优先支持上传矢量边界
- 对模糊地名提示用户确认
- 结果页显示 AOI 预览信息

### 18.3 性能风险

风险：
大区域或长时间任务执行过慢。

应对：

- 在 MVP 中限制 AOI 和时间范围
- 明确超限提示
- 后续再扩展更大规模后端

### 18.4 用户信任风险

风险：
用户拿到图但不知道是否可信。

应对：

- 输出数据选择理由
- 输出方法说明
- 输出局限性提示
- 保存任务记录，便于复核

### 18.5 范围失控风险

风险：
团队在第一版就试图接入太多指数、太多数据源、太多交互。

应对：

- 严格坚持 MVP 只做 NDVI 主链路
- 任何新功能必须证明不会明显扩大测试面
- 先以内测数据验证产品价值

## 19. 附录

### 19.1 术语表

| 术语         | 说明                                                   |
| ------------ | ------------------------------------------------------ |
| AOI          | Area of Interest，研究区                               |
| NDVI         | Normalized Difference Vegetation Index，归一化植被指数 |
| GeoTIFF      | 带空间参考信息的栅格文件格式                           |
| 合成         | 多景影像在时间窗口内按规则聚合                         |
| 有效像元比例 | 非云、非 nodata、可用于计算的像元占比                  |

### 19.2 示例方法说明模板

```md
本次分析以研究区边界为范围，选取 2024 年 6 月至 8 月的遥感影像进行 NDVI 计算。系统比较了 Sentinel-2、Landsat 8/9 和 HLS 三类数据后，最终选择 Sentinel-2 作为主要数据源，原因是其空间分辨率更适合本研究区局部地表差异识别，且该时段可用场景数量满足分析需求。处理流程包括影像筛选、云掩膜、AOI 裁剪、NDVI 计算和时间窗口中位数合成。NDVI 用于表征植被活跃程度，值越高通常表示植被覆盖越好或长势越强。结果受时段选择、云量过滤和合成策略影响，不能直接视为单日地表状态。
```

### 19.3 示例结果摘要模板

```md
### 任务摘要
- 研究区：北京西山
- 时间范围：2024-06-01 至 2024-08-31
- 指标：NDVI
- 推荐数据：Sentinel-2

### 结果解读
研究区中高 NDVI 区域主要分布在林地和连续植被覆盖较好的山区斜坡，低 NDVI 区域主要出现在建设用地、裸地和局部低覆盖区域。由于本次结果采用时间窗口中位数合成，图像反映的是该季节整体植被状况，而非某一天的瞬时状态。

### 局限性
- 结果受云量过滤影响。
- 季节合成会平滑短期波动。
- 不同数据源之间存在固有差异，本次未做多源融合比较。
```

### 19.4 MVP 结论

本项目的 MVP 不追求做成全能 GIS 平台，而是要验证一条具体、清晰、可复现的产品主链路：

> 用户通过自然语言提出植被分析任务，系统自动识别研究区与时间、比较可用遥感数据、执行 NDVI 计算、输出地图与 GeoTIFF，并用研究者能接受的方式解释结果和方法。

如果该链路在真实用户场景中被证明有价值，再进入下一阶段，逐步扩展到更多指数、更多时序分析和更复杂的科研工作流。

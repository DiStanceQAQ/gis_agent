# GIS Agent MVP PRD（后端优先版）

> 文档类型：产品需求文档（PRD）  
> 产品阶段：MVP Demo  
> 文档版本：v2.2（Manual Approval + ReAct FC）  
> 更新时间：2026-04-04

---

## 1. 文档目的与范围

本文档定义 GIS Agent MVP Demo 的后端目标、范围、功能需求、验收标准与边界。  
本版只覆盖后端，不覆盖前端交互与页面细节。

---

## 2. 产品一句话定义

GIS Agent 是一个面向遥感分析的后端执行系统：用户输入研究需求后，LLM 先理解数据和任务并生成结构化处理计划，用户手动审批（可改、可拒绝）后，系统按 ReAct + Function Calling 逐步调用 GIS 工具完成处理并输出可下载、可解释、可复核结果。

---

## 3. MVP 目标与成功标准

### 3.1 MVP 目标

1. 用户通过自然语言发起遥感/地理信息处理任务，无需手工拼接流程。  
2. 系统完成：任务解析、计划生成、人工审批、逐步执行、结果发布。  
3. 执行阶段每一步都采用 ReAct 决策并通过 Function Calling 调用白名单 GIS 工具。  
4. 输出包含数据结果、方法说明、推荐理由、风险提示。  
5. 执行过程全链路可追踪（计划、步骤、事件、产物、失败原因、LLM 调用日志）。

### 3.2 MVP 成功标准（必须全部满足）

1. 用户可对 LLM 计划执行“直接批准 / 修改后批准 / 拒绝执行”。  
2. 仅审批通过的任务才可执行；拒绝任务进入 `cancelled` 且不执行。  
3. 执行中每个计划步骤都有 ReAct 决策记录与工具调用记录。  
4. 系统在固定测试集上稳定通过，关键失败可复现、可定位。  
5. 不允许“空计划成功”：没有有效处理步骤的任务不能产出成功结论。

---

## 4. 核心原则

1. **LLM 主导决策**：解析、规划、步骤内决策、推荐由 LLM 输出结构化结果。  
2. **Human-in-the-loop**：执行前必须人工审批，支持修改计划和拒绝执行。  
3. **ReAct + Function Calling**：每一步先决策再调工具，不直接硬编码顺序动作。  
4. **结构化与可审计**：LLM 输出必须 schema 校验；每步可追踪。  
5. **执行真实性优先**：禁止占位产物冒充成功结果。

---

## 5. MVP 后端功能范围

### 5.1 输入能力

1. 自然语言任务描述（中文优先）。  
2. AOI 输入：`bbox`、`GeoJSON`、`shp/shp.zip`、已配置地名别名。  
3. 追问输入：时间切换、数据源偏好切换、重跑与重新导出。

### 5.2 数据与分析能力

1. 候选数据搜索：优先 live catalog，必要时受控 fallback。  
2. 分析类型：`NDVI`、`NDWI`、`BAND_MATH`、`FILTER`、`SLOPE_ASPECT`、`BUFFER`。  
3. 处理链：支持栅格/矢量基础操作及混合链路执行。  
4. 操作白名单必须与后端 operation registry 一致。

### 5.3 输出能力

1. 数据下载：`GeoTIFF`、`GeoJSON`、`GPKG`、`Shapefile`、`CSV`。  
2. 文本解释：`methods.md`、`summary.md`。  
3. 推荐输出：主推/备选、理由、风险提示、置信度。  
4. 产物元数据：类型、投影、尺寸/要素统计、校验和、来源步骤、时间戳。

### 5.4 执行策略能力

1. 默认执行策略为 `manual_approval`。  
2. 计划状态：`draft -> approved -> running` 或 `draft -> rejected/cancelled`。  
3. 审批动作：直接批准、修改后批准、拒绝执行。  
4. 拒绝执行后任务不可进入执行队列。

---

## 6. 端到端流程（后端）

1. 用户提交需求/上传 AOI。  
2. LLM 解析任务，输出结构化 `TaskSpec`。  
3. LLM 规划任务，输出结构化 `TaskPlan` 与 `OperationPlan`。  
4. 用户审阅计划，可编辑后提交批准，或拒绝执行。  
5. 若拒绝：任务写入 `cancelled` 与拒绝原因，流程结束。  
6. 若批准：Worker（LangGraph）进入执行循环。  
7. 每个步骤执行 ReAct：判断是否继续、选择函数、调用 GIS 工具、写 observation。  
8. 生成并发布产物与解释文本。  
9. 用户可基于当前任务追问重跑。  
10. 全链路日志（含 LLM 调用）可按 `task_id` 追溯。

## 7. 功能需求（FR）

### FR-01 任务创建与会话管理

1. 支持创建会话、发送消息、生成任务。  
2. 支持按会话查看最近任务。  
3. 支持失败任务重跑与参数覆盖重跑。

### FR-02 LLM 任务解析

1. 抽取 AOI、时间范围、分析类型、输出偏好。  
2. 输入不完整时返回澄清要求。  
3. 解析结果必须经过 schema 校验。

### FR-03 LLM 任务规划

1. 生成带步骤、依赖、目标、状态的 `TaskPlan`。  
2. 生成可编辑的 `OperationPlan`。  
3. 计划持久化到任务记录。  
4. 计划错误需返回明确错误码与错误信息。

### FR-04 人工审批与拒绝执行

1. 任务创建后默认进入 `awaiting_approval`。  
2. 支持计划编辑后批准执行。  
3. 支持拒绝执行并写入 `cancelled`。  
4. 拒绝后不可执行，且保留拒绝事件与原因。

### FR-05 步骤执行（ReAct + Function Calling）

1. 每个计划步骤都要执行 ReAct 回合：`reason -> decide -> act -> observe`。  
2. act 必须通过 Function Calling 调用白名单 GIS 工具。  
3. 每步写入 `tool_execution_started/completed/failed` 与 ReAct observation。  
4. 单步 ReAct 必须受回合数、超时、重试预算保护。

### FR-06 Operation Plan 真实性约束

1. `OperationPlan` 必须通过 schema 与依赖校验。  
2. `OperationPlan.nodes` 不能为空。  
3. 操作名白名单必须覆盖后端已支持操作并与 registry 一致。  
4. 禁止用占位产物作为“成功执行”依据。

### FR-07 候选搜索与推荐

1. 自动搜索候选并返回可比较摘要。  
2. LLM 输出主推/备选及理由、风险提示。  
3. 推荐结果持久化并可追溯来源。

### FR-08 产物发布

1. 生成结果数据、方法说明、结果摘要。  
2. 产物落库存储并提供下载接口。  
3. 产物元数据可查询且字段完整。

### FR-09 追问与重跑

1. 支持同会话上下文继承后重跑。  
2. 支持快捷重跑（换时间、换数据源、重新导出）。  
3. 新任务保留与父任务关系。

### FR-10 可观测与审计

1. 记录 LLM 调用元数据（模型、耗时、token、request_id）。  
2. 记录 parse/plan/recommend/step-react 各阶段日志并关联 `task_id`。  
3. 记录执行链路（计划、步骤、事件、fallback、错误、拒绝原因）。  
4. 支持定位“哪一步、为什么失败、为何拒绝、是否 fallback、实际执行参数”。

---

## 8. 非功能要求（NFR）

### NFR-01 稳定性

1. 核心测试集回归通过率稳定。  
2. 关键异常场景可重现、可定位。

### NFR-02 可解释性

1. 方法说明、推荐理由、fallback 必须来自执行记录。  
2. 禁止返回无法追溯来源的自由文本结论。

### NFR-03 性能

1. 任务进度可实时拉取。  
2. 中小 AOI 典型任务耗时可接受（以性能基线跟踪）。

### NFR-04 安全与边界

1. LLM 输出必须结构化校验。  
2. 仅允许白名单工具与操作。  
3. 计划与运行时均执行边界保护（steps/tool_calls/react_turns/timeout/retry）。

---

## 9. 验收标准（UAT）

### UAT-01 审批执行成功路径

1. 任务创建后进入 `awaiting_approval`，审批前不执行。  
2. 用户可直接批准或修改后批准；批准后任务进入执行。  
3. 任务详情可看到完整计划、步骤、推荐、产物与解释文本。

### UAT-02 拒绝执行路径

1. 用户可拒绝执行任务。  
2. 任务状态变为 `cancelled`，不会进入执行队列。  
3. 事件流包含拒绝动作与拒绝原因。

### UAT-03 ReAct + Function Calling 路径

1. 每个步骤均有 ReAct 决策记录。  
2. 每个步骤的工具调用来源于 Function Calling。  
3. 失败时有 `failed_step`、`error_code`、错误详情。

### UAT-04 空计划防护

1. 空 `operation_plan.nodes` 不得执行为成功。  
2. 必须返回结构化错误并可在事件流中定位。  
3. 不允许产出占位成功产物。

### UAT-05 追问路径

1. 支持同会话内变更时间窗/数据偏好后重跑。  
2. 上下文继承正确，变更项可见。  
3. 新任务与父任务关系可追踪。

---

## 10. 风险与应对

1. **LLM 输出不稳定**：结构化校验 + repair retry + 明确错误码。  
2. **ReAct 回合成本上升**：设置单步回合预算与超时。  
3. **外部数据源波动**：受控 fallback，显式记录执行语义与风险。  
4. **执行复杂度上升**：强化状态机、事件流、审计字段。  
5. **假成功风险**：强制非空计划校验，禁止占位结果冒充成功。

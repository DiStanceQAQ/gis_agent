# GIS Agent MVP PRD

> 文档类型：产品需求文档（PRD）  
> 产品阶段：MVP  
> 文档版本：v2.0
> 更新时间：2026-04-02

---

## 1. 文档目的

本文档定义 GIS Agent MVP 的产品目标、范围、功能需求、验收标准与上线边界。  

---

## 2. 产品一句话定义

GIS Agent 是一个面向遥感分析的 GIS 工作台：用户输入研究需求后，系统由 LLM 生成执行计划并自动调用 GIS 工具链，最终输出可下载、可解释、可复核的数据结果。

---

## 3. MVP 目标与成功标准

### 3.1 MVP 目标

1. 用户通过自然语言发起 遥感/地理信息数据处理 任务，无需手工拼接遥感处理流程。  
2. 系统自动完成：任务理解、计划生成、工具执行、结果发布。  
3. 输出不只是图像，还包含数据选择理由、方法说明、风险提示。  
4. 任务过程全链路可追踪（计划、步骤、事件、产物、失败原因）。

### 3.2 MVP 成功标准（必须全部满足）

1. 一个普通用户无需手工处理原始遥感数据，也能完成一次标准 遥感/地理信息数据 分析。  
2. 结果不是“黑盒图像”，而是带有数据选择理由、方法说明和局限性提示的研究输出。  
3. 系统在固定测试集上具有稳定成功率，不依赖人工临时干预。

---

## 4. 目标用户与典型场景

### 5.1 目标用户

1. GIS / 遥感初学者：能提研究问题，但不会完整流程。  
2. 科研人员：会基础操作，但希望减少重复劳动。  
3. 行业分析人员：要结果可解释、可复核、可下载。

### 5.2 典型场景

1. “帮我计算 2024 年夏季某区域 NDVI，导出图和 GeoTIFF。”  
2. “Sentinel-2 和 Landsat 哪个更适合这个任务？”  
3. “换成 2023 年重算，并给我方法说明。”

---

## 5. 产品定位与设计原则

### 6.1 产品定位

这是 GIS 工作台，不是聊天机器人。  
地图与结果工作区是主视图，任务流面板负责展示 Agent 执行过程。

### 6.2 核心原则

1. **LLM 主导决策**：任务解析、计划生成、数据方案排序由 LLM 输出。  
2. **Plan-and-Execute**：先有显式计划，再按步骤执行工具。  
3. **结构化与可审计**：LLM 输出必须结构化校验；所有步骤可追踪。  
4. **GIS 处理可复现**：实际栅格处理通过后端工具链执行并持久化产物。

---

## 6. MVP 功能范围

### 7.1 输入能力

1. 自然语言任务描述（中文优先）。  
2. AOI 输入：`bbox`、`GeoJSON`、`shp`、已配置地名别名。  
3. 追问输入：时间切换、数据源偏好切换、重跑与重新导出。

### 7.2 数据与分析能力

1. 数据源：如果用户没指定，就让LLM通过搜索确定，给用户选择。  
2. 分析类型：常见的遥感地理空间数据处理，如波段运算（NDVI、NDWI）、滤波、计算坡度坡向、缓冲区等。  

### 7.3 输出能力

1. 处理过后的数据下载。  
2. 方法说明（Markdown）。  
3. 结果摘要与局限性提示。  
4. 候选数据源比较与推荐理由。

### 7.4 界面能力（GIS 工作台）

1. 顶部导航。  
2. 左侧结果与图层面板（图层显隐、透明度、下载入口）。  
3. 中央 OpenLayers 地图（缩放、全屏、比例尺、坐标显示）。  
4. 右侧任务面板（任务输入、AOI 上传、参数卡、步骤时间线、候选比较、追问按钮）。

---

## 7. 端到端流程（MVP）

1. 用户提交需求 / 上传 AOI。  
2. LLM 解析任务，输出结构化 `TaskSpec`。  
3. LLM 规划任务，输出结构化 `TaskPlan`。  
4. Worker 进入执行循环，按 `plan.steps` 调用工具。  
5. 每步写入 `tool_execution_started/completed` 事件和 observation。  
6. 生成产物并发布到工作台。  
7. 用户可基于当前任务继续追问重跑。

---

## 8. 功能需求（FR）

### FR-01 任务创建与会话管理

1. 支持创建会话、发送消息、生成任务。  
2. 支持按会话查看最近任务。  
3. 支持失败任务重跑。

### FR-02 LLM 任务解析

1. 从输入中抽取 AOI、时间范围、分析类型、输出偏好。  
2. 对不完整输入返回澄清要求。  
3. 解析结果必须通过后端 schema 校验。

### FR-03 LLM 任务规划

1. 生成带步骤、依赖、目标、状态的 `TaskPlan`。  
2. 计划持久化到任务记录。  
3. 前端可直接渲染计划与执行状态。

### FR-04 执行循环

1. 按计划逐步执行，不允许跳过状态写入。  
2. 每步都有 started/completed/failed 事件。  
3. 失败时记录失败步骤、错误码、错误信息。

### FR-05 候选搜索与推荐

1. 自动搜索候选数据并返回可比较摘要。  
2. LLM 输出主推/备选及理由、风险提示。  
3. 推荐结果可持久化并返回前端。

### FR-06 产物发布

1. 生成 结果数据、方法说明、结果摘要。  
2. 产物落库存储并提供下载接口。  
3. 结果页可完整消费全部产物。

### FR-07 追问与重跑

1. 支持基于最近任务上下文覆盖参数重跑。  
2. 支持快捷操作：换时间、换数据源、重新导出。  
3. 新任务应保留与父任务关系。

### FR-08 可观测与审计

1. 记录 LLM 调用元数据（模型、耗时、token、请求 ID）。  
2. 记录执行链路（计划、步骤、事件、fallback、错误）。  
3. 支持定位“哪一步、为什么失败”。

---

## 9. 非功能要求（NFR）

### NFR-01 稳定性

1. 核心测试集回归通过率需稳定。  
2. 关键异常场景必须可重现、可定位。

### NFR-02 可解释性

1. 方法说明、推荐理由、fallback 必须来自执行记录。  
2. 禁止返回“无法追溯来源”的自由文本结论。

### NFR-03 性能

1. 前端可实时看到任务进度。  
2. 中小 AOI 的典型任务耗时应可接受（以性能基线文档跟踪）。

### NFR-04 安全与边界

1. LLM 输出必须结构化校验。  
2. 仅允许调用白名单工具。  
3. 限制最大步骤数、最大工具调用数、超时与重试次数。

---

## 10. 验收标准（UAT）

### UAT-01 成功路径

1. 任务可从输入自动执行到产物发布。  
2. 前端可看到完整计划、步骤时间线、候选比较和产物下载。  
3. 方法说明和摘要与执行记录一致。

### UAT-02 失败路径

1. 失败任务有明确 `error_code`、`failed_step`、错误详情。  
2. fallback 触发后可见策略、原因、实际执行参数。  
3. 用户可一键重跑或调整参数重跑。

### UAT-03 追问路径

1. 支持在同会话中变更时间窗/数据偏好后重跑。  
2. 上下文继承正确，且变更项可见。  
3. 新任务与父任务关系可追踪。

---

## 11. 风险与应对

1. **LLM 输出不稳定**：通过结构化校验、重试和兜底错误码治理。  
2. **外部数据源波动**：保留 fallback，显式记录实际执行语义。  
3. **执行链路复杂度上升**：强化事件流、任务状态机和审计字段。  
4. **成本不可控**：记录 token 与模型耗时，按任务维度做成本追踪。

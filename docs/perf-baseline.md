# GIS Agent 性能基线

更新时间：2026-04-02

这份文档记录当前本地 Docker 开发环境下的典型任务耗时，用于后续观察回归趋势。这里的数字不是 SLA，只是当前工程状态的基线样本。

## 环境说明

- 运行方式：`docker compose up -d`
- 关键服务：`postgres / redis / api / worker / web`
- 数据库来源：直接查询当前开发数据库中的已完成任务与产物记录
- 典型 AOI：`bbox(116.1,39.8,116.5,40.1)`

## 基线样本

| 任务 ID | 模式 | 数据源 | 时间窗 | 影像数 | 输出网格 | 耗时 | GeoTIFF | PNG |
| --- | --- | --- | --- | ---: | --- | ---: | ---: | ---: |
| `task_edcc325c1cd3` | mock | Sentinel-2 | 2024-06-01 至 2024-06-30 | 0 | 96x96 / EPSG:4326 | `<1s` | 37,278 B | 3,557 B |
| `task_1c93f79765d5` | real | Landsat 8/9 | 2024-08-30 至 2024-08-30 | 2 | 256x256 / EPSG:32650 | `93s` | 262,714 B | 84,282 B |
| `task_cb0512357d6a` | real | Landsat 8/9 | 2024-08-22 至 2024-08-30 | 4 | 512x512 | `265s` | 1,049,728 B | 316,077 B |

## 样本解读

- `task_edcc325c1cd3` 是当前 Docker 默认异步 mock 链路的控制组。`duration_seconds = 0` 代表整型秒级统计下小于 1 秒，不代表绝对零耗时。
- `task_1c93f79765d5` 是较小真实任务，`run_ndvi_pipeline` 使用 2 景 Landsat，`valid_pixel_ratio = 0.9999`，当前更适合作为“主链可用性”回归样本。
- `task_cb0512357d6a` 是更重的真实任务，`run_ndvi_pipeline` 使用 4 景 Landsat，`valid_pixel_ratio = 0.9998`，更适合作为“真实链路性能退化”观察样本。

## 当前观察

- mock 控制组已经足够快，当前瓶颈不在编排或导出，而在真实 STAC 检索和像元级处理。
- 真实链路对景数和输出网格尺寸比较敏感，2 景到 4 景的样本从 `93s` 增长到 `265s`，接近 3 倍。
- 真实任务当前仍然适合异步执行，不适合重新切回同步 HTTP 请求。
- 基于当前样本，Docker 开发默认继续保持 `GIS_AGENT_REAL_PIPELINE_ENABLED=false` 更稳，真实链路建议按需开启。

## 建议的回归阈值

- mock 控制组如果稳定超过 `3s`，需要排查任务排队、数据库写入或导出退化。
- 2 景真实任务如果稳定超过 `120s`，需要优先检查 STAC 检索、窗口读取和栅格重投影。
- 4 景真实任务如果稳定超过 `300s`，需要重点检查 `max_items`、输出网格尺寸和 QA 掩膜阶段。

## 后续补充建议

- 增加一个 Sentinel-2 真实样本，避免性能基线只覆盖 Landsat。
- 把任务输入、景数、输出尺寸和耗时整理成固定脚本，避免完全依赖手工挑选数据库样本。
- 后续如果引入 MinIO 产物存储，需要再补一次“本地文件系统 vs 对象存储”的导出耗时对比。
